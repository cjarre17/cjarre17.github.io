Key Concepts

  1. C10K Problem: Handling 10,000+ concurrent connections on a single server.
  2. Event Loop: Non-blocking I/O model where a worker handles many sockets asynchronously.
  3. Master/Worker: Master manages config & processes; workers do the request handling.
  4. Reverse Proxy: Routes, caches, and secures traffic to internal services.
  5. TLS Termination: Nginx decrypts/encrypts HTTPS at the edge; backends can use HTTP.
  6. Caching: Serving frequently requested responses directly from Nginx.
  7. Load Balancing: Split traffic across instances to improve availability & scale.
  8. Modules: Extend Nginx functionality (e.g., security, monitoring, protocols).
  9. Configuration: Declarative syntax to define server behavior, routing, and more.
  10. High Availability: Techniques like failover and health checks to ensure uptime.
  11. Mail Server: Nginx can be configured to handle email traffic (SMTP, POP3, IMAP).

NGINX vs. Apache

A common question is how NGINX compares to the Apache HTTP Server. While both are capable, they excel in different areas:

  • Architecture: Apache uses a process-per-connection or thread-per-connection model, while NGINX uses an event-driven, asynchronous model. This makes NGINX generally faster and less memory-intensive under high load.
  • Configuration: Apache allows decentralized configuration via .htaccess files, which is convenient for shared hosting but hurts performance. NGINX centralizes config for speed and does not support .htaccess.
  • Content: NGINX is often faster at serving static files. For dynamic content (like PHP), Apache can process it internally (mod_php), whereas NGINX proxies it to an external processor (PHP-FPM).
