NGINX Architecture

NGINX is built around one master process and many other worker processes. Worker processes are event-driven and can handle many connections at once through the use of non-blocking I/O (event loops).

Master vs. Worker Role

  • Master: reads config, binds ports, manages worker lifecycle.
  • Worker: performs the actual request processing

Why Event Driven?

Compared to a “one thread per connection” model, event-driven workers avoid massive memory and context-switch overhead when handling thousands of clients.

How the Event Loop Works

The event loop is the engine that drives NGINX's efficiency. Instead of dedicating a full OS thread to each user, a worker process cycles through all active connections, handling small chunks of work as they become available.

  1. Wait: The worker waits for events on network sockets ("new connection" or "data ready").
  2. Accept: When a new client connects, the worker accepts the connection and adds it to the monitoring list.
  3. Process: When data arrives, the worker reads the request, processes it, and prepares a response.
  4. Non-Blocking: If an operation would take time (like waiting for a database), the worker doesn't wait. It moves to the next connection and checks back later.

Process Management Signals

Administrators communicate with the Master process using system signals. This allows for maintenance without downtime.

  • SIGHUP: Reload configuration. The master starts new workers with the new config and gracefully shuts down old ones.
  • SIGQUIT: Graceful shutdown. Workers finish their current requests before exiting.
  • SIGUSR1: Re-open log files (useful for log rotation).

Tip: By default, Nginx spawns about one worker per CPU core — a sensible starting point.
