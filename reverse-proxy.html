Reverse Proxy â€” What & Why

A reverse proxy sits in front of application servers and routes client requests to the appropriate backend, adding performance and security features along the way.

Core Capabilities

  • Security: expose only the proxy to the internet; hide backends.
  • Caching: serve popular content from cache to reduce load.
  • TLS Termination: offload HTTPS encryption/decryption.
  • Compression/Optimization: reduce bandwidth per response.
  • Routing & Rewrites: direct to services, rewrite URLs as needed.

Use-Case Examples

Load Balancing

Distribute traffic across multiple instances; enable automatic failover.

API Gateway

Route /users to the Users service, /orders to the Orders service, etc.

Legacy HTTPS Front

Keep an HTTP-only legacy app behind NGINX; serve it securely over HTTPS.

Static Caching

Cache blog posts or media for faster delivery and lower backend costs.

Basic Configuration Example

A simple reverse proxy configuration might look like this. It listens on port 80 and forwards all traffic to a backend application running on port 3000.

Example:
server { listen 80; server_name example.com; location / { proxy_pass http://localhost:3000; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; } }

Load Balancing Strategies

When distributing traffic, NGINX can use several algorithms to decide which server gets the next request:

  • Round Robin: Requests are distributed sequentially to the list of servers. This is the default.
  • Least Connections: The request is sent to the server with the fewest active connections, useful when request times vary.
  • IP Hash: The client's IP address is used to determine the server, ensuring a user always reaches the same backend (session persistence).
